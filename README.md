# HSE_BANK
## Функционал приложения:
	1. Импорт/Экспорт Данных (BankAccount, Category, Operation) из/в JSON, CSV, YAML форматов.
	2. Перерасчет значений счетов для аккаунтов. (Проиводится отметка обработанных операций)
	3. Реализовано меню, для доступа к всему функционалу приложения.
	4. Реализован фасад аналитики, позволяющий получить общую сводку данных банка.

## Паттерны GoF
### Поведенческие паттерны
	1. Шаблонный метод
		IFileHandler void Save<T>(IEnumerable<T> obj, string savePath)
		IFileHandler IEnumerable<T> Load<T>(string loadPAth)
		Определяет скелет алгоритма считыванивая и вывода данных, подлежащих сериализации. 
		Переропределение для данных методов не потребовалось. Их выполнение не менялось от фактических параметров. Менялась только "сигнатура" 
	2. Структура
		Ее можно наблюдать на примере абсракций :
			public abstract class IBankAccount : IUniqueId, INamed
			public abstract class ICategory : IUniqueId, INamed
			public abstract class IOperation : IUniqueId, IHasDate
			Где IUniqueId абстрактный класс,содержащий поле Guid Id, позволяющие хранить уникальные записи дял данных.
	3. Команда 
		В рамках ProxiMenu данные, возвращаемые экземпляром, были обернуты в Enum для удобства.
		В результате ответы было проще интерпретировать, что позволило реализовать паттерн Посетителя.
	4. Посетитель
		Этот паттерн был реализован в методе App.Solution(). Реализация метода посещает разлличные сервисы, формируя ответ.
	5. Итератор
		Паттерн реализован в Storage, позволяя иметь доступ к элементам хранлища по их UUID, а не по средством списка.
	6. Посредник
		Посреднико можно назвать Processor. Он инкапуслирует взаимодействие всех объектов между собой, позволяя актуализовать данные.
	7. Состояние
		Состояние приложения поддерживается и хранится в App. В зависимости от этого состояние выполняются различные действия.
### Структурные паттерны
	1. Прокси
		ProxiMenu - заместитель (посредник) между Пользовательским Menu и логикой программы. 
		Он обрабатывает команды пользователя, и переводит их в понмиаемый программе вид. 
	2. Фасад
		AnalyticsFacade - предоставляет простой доступ к аналитике.
		App выступает общим - глобальным фасадом для пользователя. Упрощая взаимодействие с программой до элемнетарных дейтсвий.
	3. Декоратор
		Позволяет классам сериализоваться библиотекам. Блокирует некоторые поля дял сериализациии. Реализован в классах, отображающих данные.
### Порождающие паттерны
	1. Реализованы Фабрики и Абстрактные фабрики в соответсвующих папках.
	2. Реализованы Сингелоты через DI-контейнер.
## Принципы GRASP и SOLID
	1.	High Cohesion || Single Responsibility Principle
		Каждвй класс отвечает только за свой функционал. Изменение реализаций других классов не влечет изменения в других классах.
		Это реализовано посредством интерфейсов и абстрактных классов.
	2. Liskov Substitution Principle
		Реализован через абстрактные классы и интерфесы, позволяя подставлять их в реализации не ломая функционал.
	3. Interface Segregation Principle
		Заметен в разделении интрфейсов и абстрактных классов для BankAcoount, Operation и т.п. (Разделение INamed, IUniqueId, IHasDate)
	4. Dependency Inversion Principle
		Абстракции не зависят от реализаций вообще.
